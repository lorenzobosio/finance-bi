<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser (Google OAuth) -> Frontend Server | Untrusted identity; only allowlisted emails may proceed |
| middleware -> protected pages | Every route gated by validated session + allowlist |
| Server Component -> Postgres | Read runs under the user JWT so RLS enforces the allowlist |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-08 | Spoofing | Forged/expired session passes the gate | mitigate | middleware uses `getUser()` (network-validated), never `getSession()` (Pitfall 3) |
| T-00-09 | Elevation of Privilege | Non-allowlisted Google account reaches the app | mitigate | middleware `isAllowed()` block + sign-out (D-13) AND the Plan-02 RLS zero-rows policy (defense-in-depth) |
| T-00-10 | Tampering | Open redirect via OAuth `redirectTo` | mitigate | `redirectTo` fixed to `${location.origin}/auth/callback`; authorized redirect URIs whitelisted in Supabase + Google Console (Plan 01 PROVISIONING) |
| T-00-11 | Information Disclosure | Drizzle (RLS-bypassing) client used for the protected-page read | mitigate | protected page reads via `@supabase/ssr` only; verify asserts no `@/lib/db` import in the page (Pitfall 1) |
| T-00-12 | Spoofing | exchangeCodeForSession fails to write cookies (v2.91.0) | mitigate | await cookie persistence in the callback; confirm the pinned supabase-js version is unaffected; manual post-login smoke (Pitfall 5) |
</threat_model>
