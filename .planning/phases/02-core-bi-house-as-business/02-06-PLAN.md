<objective>
Ship the Transações write-path vertical slice — the heart of CAT-04/05 (CONTEXT slice 2.4). After this plan a user can view the transactions table, re-categorize a transaction, assign its cost center, create a forward-only rule from it (so they never re-tag the same merchant twice), and explicitly re-apply a rule to past rows with an idempotent affected-count — without raw history ever being silently rewritten.

This completes the user-editable rules loop on top of the DB-backed engine (Plan 02). All mutations are Server Actions: validated by zod, written through @supabase/ssr under the 2-email allowlist RLS, never service_role (RESEARCH Pattern 6/7, Pitfall 3). The edit-one-row default + the separate explicit re-apply enforce CAT-05/D2-03.

Purpose: CAT-04 (view table, re-categorize, create rule, assign cost center), CAT-05 (re-apply is explicit + idempotent; raw history never silently rewritten), CAT-06 (excluded rows visibly tagged so the table reconciles with the P&L).
Output: the Transações page, the table + edit popover components, three Server Actions, and the recategorize/reapply tests.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser edit popover → Server Action | The recategorize/create-rule/re-apply edit crosses from untrusted client into 'use server' functions; zod validates before any DB write |
| Server Action (WRITE plane) → transactions/rules | Writes through @supabase/ssr (anon + user JWT); the existing allowlist RLS on transactions/rules authorizes — no service_role |
| RSC page (READ plane) → transactions | supabase.from('transactions') under the user JWT; RLS enforces the 2-email allowlist (shared visibility — both users may edit any row, cost center is a label not an access wall) |
| developer → public repo | page + components + actions committed publicly — no PII (merchant names in fixtures are synthetic), no service_role/DATABASE_URL |
